<threat_model>
## Trust Boundaries
- `TB-01`: Internet client -> NestJS HTTP entrypoints (`/api/v1/*`), including malformed payloads and hostile headers.
- `TB-02`: Runtime environment variables -> validated process config (`ConfigModule` + env parser).
- `TB-03`: API health/readiness response -> deployment orchestrator and future database adapter.

## STRIDE Register
| threat_id | category | component | disposition | mitigation_plan |
| --- | --- | --- | --- | --- |
| T-01 | Tampering | Validation pipeline and DTO error payloads | mitigate | Ensure ValidationPipe and exception filter emit sanitized, deterministic envelopes; add tests asserting no raw target/value leakage. |
| T-02 | Information Disclosure | Global exception filter | mitigate | Normalize framework/runtime exceptions to bounded `error.code/message/details`; include request metadata only, never stack traces or secrets. |
| T-03 | Spoofing | Request correlation/logging | mitigate | Treat `x-request-id` as optional untrusted input; generate/fallback in middleware and log canonical request id field for traceability. |
| T-04 | Denial of Service | Health/readiness endpoint | mitigate | Keep health handler lightweight and public; return stub-compatible readiness object without blocking I/O in foundation phase. |
| T-05 | Repudiation | Startup and request lifecycle logs | mitigate | Require structured startup/request logs with stable keys so failures are auditable in Railway/local collectors. |
</threat_model>
